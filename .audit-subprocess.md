## 总体评分: 5/10

架构设计考虑了进程组隔离和优雅终止，但存在 **关键功能未启动**、**竞态条件** 和 **资源泄漏** 问题。两处 P0 将严重影响生产可用性。

## P0 — 必须修复

### P0.1 — 健康检查守护循环未启动

**文件:** `serve_backend.go:490`  
**函数:** `startServeHealthCheck()`  

`startServeHealthCheck()` 定义了完整的 60s 周期性健康检查逻辑（检测 serve 进程是否存活、自动重启、指数退避），但在 `main()` 中**从未被调用**。任何地方都没有调用它。

**故障场景:** serve 进程因 OOM/SIGKILL/crash 异常退出后，桥接不会自动恢复。直到用户发送下一条消息触发 `ensureServe()` 才能发现并重启。在多用户场景下，部分用户可能数小时无响应却无人察觉。

**修复建议:** 在 `main.go:465` 附近添加 `app.startServeHealthCheck()` 调用。

### P0.2 — 端口分配无冲突避免，且 chatID 参数未使用

**文件:** `serve_backend.go:28-33`  
**函数:** `portForChat(chatID int64) int`

```go
func portForChat(chatID int64) int {
    const base = 18780
    const span = 8000
    p := base + (int(portSeq.Add(1)) % span)
    return p
}
```

两个问题：

1. **chatID 参数被忽略** — 端口按全局计数器轮询分配，同一 chat 在不同时间调用 `portForChat` 会得到不同端口。但 `session.servePort` 在 session 创建时就固定了，所以重启后若从 state.json 恢复则使用旧端口，否则新分配的端口可能与另一个 session 冲突。

2. **无冲突检测** — 8000 个端口（18780-26779）用尽后从头开始轮询，已分配的端口可能仍被旧进程占用。`startServe` 不会检查端口可用性就直接启动子进程，`waitServeReady` 在端口被占用时会等待 45s 然后失败。

**故障场景:** ~200 个并发 chat 时碰撞概率接近 100%。碰撞后 serve 进程无法绑定端口，`waitServeReady` 超时 45s，用户收到 "Reasonix 服务启动失败"。

**修复建议:**  
- 使用 chatID 派生的确定性端口号（如 `base + (chatID % span) + span`）  
- 分配前调用 `isPortInUse()` 检测，若占用则尝试下一个可用端口  
- 对碰撞实施指数退避重试

## P1 — 应当修复

### P1.1 — killDescendants 在遍历 /proc 后可能遗漏子进程

**文件:** `utils.go:19-88`  
**函数:** `killDescendants()`

`walkTerm` 在 Phase 1 开始时通过 `os.ReadDir("/proc")` 获取一次快照。如果某个进程在 Phase 1 遍历过程中又 fork 了子进程（例如收到 SIGTERM 后 fork 清理进程），新子进程不会被访问到 — 不会收到 SIGTERM/SIGKILL。

**故障场景:** 某个 `reasonix serve` 进程的退出处理程序创建了子进程（git commit、临时文件清理等），`killDescendants` 未向该子进程发送 SIGTERM，使其成为孤儿进程被 init 收养。

**修复建议:** Phase 2 和 Phase 3 之间再扫描一次 /proc，或改用 process group 进行终止（但 `Setpgid` 已经创建了独立的进程组，用 `syscall.Kill(-pid, SIGTERM)` 以组为单位发送信号）。

### P1.2 — SSE 看门狗与 Scanner 并发访问 resp.Body

**文件:** `serve_backend.go:778-793` + `serve_backend.go:796-798`  

看门狗 goroutine 在空闲超时时调用 `resp.Body.Close()`，同时主 goroutine 正在 `sc := bufio.NewScanner(resp.Body)` → `sc.Scan()` 循环中读取同一个 body。`http.Response.Body.Close()` 不是并发安全的，会导致数据竞态（data race），可能使 scanner 读到不完整的数据或 panic。

**故障场景:** SSE 流空闲 5 分钟后，看门狗关闭 body，主 scanner goroutine 可能同时读取部分关闭的连接，触发 `read on closed pipe` 错误或未定义行为。

**修复建议:** 使用 `sync.Once` 或 `context.Context` 协调关闭。看门狗应通过 context 通知主循环关闭 body，而不是直接关闭。例如：

```go
// 看门狗检测到空闲 → watchdogCancel()
// 主循环的 ctx.Done() 分支执行 resp.Body.Close()
```

### P1.3 — isPortInUse 存在 TOCTOU 竞态

**文件:** `serve_backend.go:44-51`  

```go
func isPortInUse(port int) bool {
    ln, err := net.Listen("tcp", fmt.Sprintf("127.0.0.1:%d", port))
    if err != nil {
        return true
    }
    ln.Close()
    return false
}
```

检查后到实际使用前端口可能被其他进程占用。不过当前代码仅用于诊断日志和等待端口释放后的轮询，不作为互斥手段，风险较低。

**修复建议:** 在 `startServe` 中绑定端口前使用 `SO_REUSEADDR` + `SO_REUSEPORT` 或直接尝试 listen 并传递 listener（而非端口号）给子进程。

### P1.4 — 信号处理未直接传播到进程组

**文件:** `main.go:480-485`  

```go
case <-ctx.Done():
    app.msgWg.Wait()
    app.cancelAllTasks()
    app.waitTasksDone(5 * time.Second)
    app.stopAllServes()
```

桥接收到 SIGTERM 后通过 `stopAllServes()` 逐个停止子进程，而不是将 SIGTERM 传播给子进程组。虽然 `stopAllServes` 最终会终止所有子进程，但存在竞态窗口：信号处理中先 `cancelAllTasks()` 取消上下文，SSE 读取 goroutine 可能因上下文取消而退出并关闭连接，但 `stopAllServes` 尚未运行，导致 serve 进程的 `POST /submit` 请求被断开但 serve 进程本身未收到停止信号。

**故障场景:** 桥接收到 SIGTERM，`runServeTurn` 的 context 被取消，但 serve 子进程仍在运行（等待下一轮输入），成为孤儿进程。`stopAllServes` 最终会清理它，但中间有 5s 的 `waitTasksDone` 延迟。

**修复建议:** 在信号处理路径开头就向所有子进程组发送 SIGTERM（通过 `stopAllServes` 或更直接的循环），再等待任务完成。

## P2 — 建议修复

### P2.1 — startServe 中的 cmd.Wait goroutine 与 stopSessionServe 的 cmd.Process.Wait 竞态

**文件:** `serve_backend.go:437-447` vs `serve_backend.go:270-273`

`startServe` 启动的 goroutine 调用 `cmd.Wait()`，`stopSessionServe` 调用 `cmd.Process.Wait()`。当 `stopServe` 触发时，两个 Wait 可能同时进行，第二个 Wait 返回 "wait: no child processes" 错误（被记录但无害）。虽然不导致僵尸进程，但会产生误导性日志。

**修复建议:** 在 `stopSessionServe` 中设置一个标志（如 `s.serveCmd = nil` 前使用原子操作），让 goroutine 的 `cmd.Wait()` 知道已被接管，不再记录错误。

### P2.2 — restorePersistedSessions 并发无限制

**文件:** `serve_backend.go:596-601`

```go
go func(chatID int64) {
    if err := a.startServe(chatID, false); err != nil {
        log.Printf("chat=%d: restore serve failed ...", chatID, err)
    }
}(rec.ChatID)
```

每个持久化的 session 都启动一个 goroutine 并发调用 `startServe`。如果有 100 个 session，会同时启动 100 个 `reasonix serve` 进程和 100 个端口检查 HTTP 请求（`waitServeReady` 的轮询）。这可能触发系统 ulimit 限制或短时间内 API 限流。

**修复建议:** 使用信号量（channel based semaphore）限制并发数到 5-10。

### P2.3 — terminateProcessGroup 不回收僵尸进程

**文件:** `stale_serve.go:136-163`

`terminateProcessGroup` 通过 `syscall.Kill(pid, 0)` 轮询进程存活状态，但从不调用 `os.Wait()` 或 `syscall.Wait4()` 回收进程表项。如果进程已是孤儿（由 init 收养），init 会回收；但如果终止的进程是桥接的孙子进程（通过 `Setpgid` 成为子进程组的成员），桥接不会回收它，导致僵尸。

**故障场景:** `cleanupStaleServesOnStartup()` 调用 `terminateProcessGroup(pid, 8s)` → SIGTERM → SIGKILL。被杀的 serve 进程变成僵尸（defunct）直到桥接自身退出。

**修复建议:** 在确认进程已死后调用 `syscall.Wait4(pid, nil, 0, nil)` 回收。

### P2.4 — cron.go exec.CommandContext 未设置 Setpgid

**文件:** `cron.go:282`

```go
cmd := exec.CommandContext(ctx, a.cfg.ReasonixBin, "run", "--resume", tmpPath, "--model", "deepseek-v4-flash", "--", fullPrompt)
```

Cron 任务执行的 `reasonix run` 没有设置 `Setpgid: true`。这意味着：
- 定时任务进程与桥接在同一进程组，SIGTERM 会同时传给它们
- 没有独立的进程组，`killDescendants` 无法按组管理

对比 `serve_backend.go:418` 的 `startServe` 正确设置了 `Setpgid: true`。

**修复建议:** 添加 `cmd.SysProcAttr = &syscall.SysProcAttr{Setpgid: true, Pgid: 0}`。

### P2.5 — media.go exec.Command 未设置 Setpgid 且无超时

**文件:** `media.go:182,233`

`ffmpeg` 和 `pdftoppm` 命令使用 `cmd.Run()`（同步等待），但没有超时保护。恶意用户上传的媒体文件可能导致 ffmpeg/pdftoppm 执行时间过长（如畸形文件触发无限循环），阻塞 handleMessage goroutine。

**修复建议:** 使用 `exec.CommandContext(ctx, ...)` 配合合理的超时（如 30s）。

### P2.6 — serve 子进程无 stdout 捕获，启动失败诊断困难

**文件:** `serve_backend.go:417`

```go
cmd.Stderr = os.Stderr
cmd.Stdout 未设置（默认为 nil → /dev/null）
```

serve 进程的标准输出被丢弃，标准错误传给桥接的 stderr。如果 serve 进程因配置错误在启动初期写入 stdout 而不是 stderr，诊断信息会丢失。`waitServeReady` 仅能判断端口是否可访问，无法获取具体的启动错误。

**修复建议:** 将 stdout 也指向桥接的 stderr 或捕获到日志缓冲区：

```go
cmd.Stdout = os.Stderr  // 或者用 bytes.Buffer 捕获
```

```

  → waitTasksDone(5s)      // 等待 SSE 流结束

  → findStaleReasonixServePIDs()  // 扫描 /proc
```

## 总结

**两处 P0**（健康检查未启动、端口分配无冲突）使生产环境的自动恢复和扩展性严重受限。**P1 集中在协调 /proc 遍历遗漏子进程、SSE 看门狗数据竞争、端口检查竞态。** 整体子进程生命周期管理框架设计合理（进程组隔离、优雅终止、僵尸预防），但实现细节和边缘情况处理不足。

建议优先修复 P0.1（一行代码即可恢复自动恢复能力）和 P0.2（影响所有多会话部署）。
