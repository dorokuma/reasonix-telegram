# Deep Analysis: Bot Stuck in "Typing..." After Approval Deny

## Summary

Investigation of the sequence: Approval request → User clicks Deny → User types "stop" → Bot says "no active task" → User sends new message → Bot stuck in typing.

Root cause: Multiple interacting bugs create a scenario where the typing indicator runs forever.

---

## Finding 1: `s.task` prematurely cleared by `releaseTask()` during `onApprovalRequest`

### File: `stream.go:600-659` (onApprovalRequest callback)
### File: `stream.go:119-126` (releaseTask)
### File: `stream.go:131-262` (endStream)

When the approval_request SSE event arrives, `onApprovalRequest` is called. It calls `signalFlush()` (line 603), which triggers `endStream()` → `releaseTask()` (line 157, 166, 174, 252, 261).

`releaseTask()` at line 119-126:
```go
releaseTask := func() {
    s.mu.Lock()
    if s.task == thisTask {
        s.task = nil          // ← clears s.task BEFORE the turn is done
        s.wakePusher = nil
    }
    s.mu.Unlock()
}
```

This sets `s.task = nil` while the SSE goroutine is **still running** and the turn hasn't completed yet. The typing indicator (started at line 66-68) is still active.

### Impact: `/stop` reports "no active task"

When the user later types `/stop`, `handler.go:310-325` checks `s.task`:
```go
t := s.task
if t == nil {
    a.reply(m.Chat.ID, "当前没有进行中的回复")  // ← matches user's report
    return
}
```

Since `s.task` was already cleared, the stop command does nothing. But the SSE goroutine is still running, consuming the rest of the turn's events. The typing indicator continues.

## Finding 2: SSE sub-goroutines can leak and hang

### File: `serve_backend.go:992-1024` (runServeTurn)
### File: `serve_backend.go:1411-1450` (consumeServeEvents)

After the user's new message triggers a SECOND `runTask`, a new SSE connection is established. This new connection subscribes to the Broadcaster with `afterSeq=0` (starts from the beginning of the event log).

### File: `/root/reasonix/internal/serve/broadcaster.go:85` (Subscribe)

The Broadcaster's subscriber channel has buffer size **64**:
```go
ch := make(chan []byte, 64)
```

The ring buffer stores **10,000** events (`ringSize = 10000`).

When a subscriber connects with `afterSeq=0`, the Subscribe function replays all events from the ring buffer into the channel. But if there are more than 64 events, the channel fills up and the replay stops **prematurely** (line 112-114):
```go
select {
case ch <- entry.data:
default:
    goto doneReplay    // ← stops replay when channel is full
}
```

After this premature stop, the subscriber is added to the Broadcaster's subscriber list (line 120). But it never received the remaining events (including any `turn_done` event that might be in the ring buffer at a later index).

The subscriber then waits for new events on the channel. If the Controller is still processing the old turn (or in a `pendingReentry` state), no new events arrive. The subscriber hangs **forever**, causing `runServeTurn` to never return, causing `runTask` to never return, causing `beginTyping`'s goroutine to never be cancelled.

## Finding 3: Typing indicator has no timeout, only task-completion dependency

### File: `handler.go:586-603` (beginTyping)
### File: `stream.go:66-82`

`beginTyping()` returns a `cancel` function. This function is stored in `runningTask.stopTyping` and is only called by:
1. `runTask`'s defer `defer stopTyping()` (line 82) — only when `runTask` returns
2. `/stop` command: `t.StopTyping()` (handler.go:322) — only if `s.task != nil`

There is **no independent timeout** on the typing indicator. If `runTask` never returns (due to Finding 2), the typing indicator keeps sending `sendChatAction(typing)` every 4 seconds **forever**.

The only "timeout" is the task context's `MaxDuration` (default 120 minutes). After 2 hours, the context cancels and `runTask` eventually returns.

## Finding 4: Lost user input via `pendingReentry`

### File: `/root/reasonix/internal/control/controller.go:321-348` (runGuarded)

When the Controller is already running (processing the rest of the turn after the deny), a new POST /submit from the bridge sets `pendingReentry = true` and **discards the actual input**:

```go
func (c *Controller) Submit(input string) {
    // ...
    default:
        c.runRefTurn(input)
    }
}

func (c *Controller) runRefTurn(input string) {
    c.runGuarded(func(ctx context.Context) error { ... })
}

func (c *Controller) runGuarded(body func(ctx context.Context) error) {
    c.mu.Lock()
    if c.running {
        c.pendingReentry = true   // ← sets flag, but input is NOT stored
        c.mu.Unlock()
        return                    // ← returns without processing the input
    }
    // ...
    // When this turn finishes:
    if c.pendingReentry {
        c.Send("")                // ← sends EMPTY STRING, original input lost!
    }
}
```

The user's message is accepted by the bridge (HTTP 202 from POST /submit) but never processed by the model.

## Finding 5: Race between old SSE subscriber removal and new subscriber replay

### File: `serve_backend.go:1031-1050` (connectAndConsumeSSE)

When the old SSE connection (G1) completes, it closes the HTTP response body. The serve's `events` handler detects the disconnect and calls `unsubscribe()`. But this happens asynchronously — there's a window where the old subscriber is still in the Broadcaster's subscriber map while the new subscriber is being added.

If the old `turn_done` event is emitted by the Controller before the old subscriber is removed, it could be sent to both the old and new subscriber channels. This is not harmful per se, but combined with Finding 2's channel-buffer overflow, it increases the chance that the new subscriber misses `turn_done`.

---

## Root Cause Summary

The primary issue is **Finding 2**: the Broadcaster's subscriber channel buffer (64) is much smaller than the ring buffer (10,000). When a new SSE connection subscribes with `afterSeq=0`, it only receives up to 64 old events. If `turn_done` is among the remaining events not replayed, the subscriber hangs forever.

This is triggered by the specific sequence:
1. Approval request → `onApprovalRequest` → `releaseTask()` clears `s.task` (Finding 1)
2. User's `/stop` finds no task, does nothing (Finding 1)
3. New message → new SSE connection → replay overflow → misses `turn_done` (Finding 2)
4. SSE hangs → `runTask` hangs → typing never stops (Finding 3)

## Recommended Fixes

1. **Fix `releaseTask` in `onApprovalRequest`**: Do NOT call `endStream`/`releaseTask` during `onApprovalRequest`. The approval pause is not the end of the turn — the stream should stay open. Remove the `signalFlush()` call from `onApprovalRequest` (stream.go:603), or at least don't call `releaseTask()` inside the resulting `endStream`.

2. **Increase subscriber channel buffer**: Change `make(chan []byte, 64)` to `make(chan []byte, ringSize)` (10,000) so the entire ring buffer fits in one replay. Or implement a batched replay that sends all events.

3. **Or, better: change the replay logic**: Instead of buffering all replay events, the Broadcaster should send them in batches or the subscriber should read events in a loop during Subscribe.

4. **Add deadline to typing indicator**: `beginTyping` should have a fallback timeout (e.g., 5 minutes) that stops the typing even if the task doesn't complete normally.

5. **Store pending input**: When `runGuarded` sets `pendingReentry = true`, store the actual input so it can be processed after the current turn finishes, instead of calling `c.Send("")`.
