## P0 — 关键缺陷

### P0-1: formatMessage 后切分可能破坏 MDV2 语法 (telegram_text.go:411-415)

`sendFormattedParts` 先将文本通过 `formatMessage()` 转换为 MarkdownV2 格式，然后调用 `splitTelegramText()` 在 4096 rune 边界切分。但 split 完全不感知 MDV2 语法结构，可能在格式化标记中间切分（如把 `*bold text` 和 ` rest*` 分到两条消息），导致 Telegram API 抛出 "can't parse entities" 错误。

```go
// telegram_text.go:411-415
displayText = formatMessage(displayText)
parseMode = tgbotapi.ModeMarkdownV2
parts := splitTelegramText(displayText, telegramMaxMessageRunes)
```

虽然下游 `sendMessageParts` (telegram_text.go:487-496) 有 fallback 会 strip 格式后以纯文本重发，但格式信息完全丢失。影响所有长回复。

**影响**：长消息第二段起格式丢失；API 层面会触发一次 "can't parse entities" 错误后 fallback。

### P0-2: `capTelegramMessage` 未预留格式化转义空间 (telegram_text.go:335-350, utils.go:126-129)

`capTelegramMessage` 将文本截断到 ≤4096 rune，然后 `formatMessage()` 为其添加 MDV2 转义反斜杠（每个特殊字符加一个 `\`）。若原文刚好 4096 字符且含大量特殊字符，转义后可能超过 4096 限制。

调用链：`editCommentary`(utils.go:126) → `capTelegramMessage(text)` → `formatMessage(text)` → 直接 `sendWithRetry`(utils.go:132)，未经拆分。若格式化后 >4096，API 返回 MESSAGE_TOO_LONG，触发 plaintext fallback。

```go
// utils.go:126-132
text := capTelegramMessage(appendText)
formatted := formatMessage(text)
edit := tgbotapi.NewEditMessageText(chatID, messageID, formatted)
edit.ParseMode = tgbotapi.ModeMarkdownV2
_, err := a.sendWithRetry(edit, chatID)
```

**影响**：短文本无风险，但接近 4096 边界的文本可能因转义膨胀导致 API 拒绝，然后 fallback 到纯文本丢失格式。

## P1 — 严重缺陷

### P1-1: `_MDV2_ESCAPE_RE_RE` 死代码 (telegram_format.go:18)

`telegram_format.go` 第 18 行定义了 `_MDV2_ESCAPE_RE_RE`，但在整个项目中从未被引用。`formatMessage()` 实际使用的是 `telegram_escape.go` 中的 `_escapeMdv2()`（使用 `_MDV2_ESCAPE_RE`）。这是重复编译的浪费，且可能引起后续维护者的困惑。

### P1-2: `isReasonixNoise` 过滤过于激进 (main.go:74-113)

`reStatusDot` 正则 `^\s*·\s+\S+:` 会匹配以 `· item: description` 开头的合法内容行（如自然语言的冒号项目符号）。虽在 cron 管道输出场景中合理，但在 `appendChunk()` (serve_backend.go:1048) 中被用于流式 chunk 过滤，可能导致正常的带冒号项目符号内容被丢弃。

```go
// main.go:60
reStatusDot = regexp.MustCompile(`^\s*·\s+\S+:`)
```

### P1-3: `stripMdv2` 正则在运行时反复编译 (telegram_format.go:443-453)

```go
cleaned = regexp.MustCompile(`\\([_*\[\]()~` + "`" + `>#+\-=|{}.!\\])`).ReplaceAllString(text, `$1`)
cleaned = regexp.MustCompile(`\*\*([^*]+)\*\*`).ReplaceAllString(cleaned, `$1`)
cleaned = regexp.MustCompile(`\*([^*]+)\*`).ReplaceAllString(cleaned, `$1`)
cleaned = regexp.MustCompile(`_([^_\n]+)_`).ReplaceAllString(cleaned, `$1`)
cleaned = regexp.MustCompile(`~([^~]+)~`).ReplaceAllString(cleaned, `$1`)
```

每次调用 `stripMdv2` 都重新编译 5 个正则表达式。应提升为包级变量。在频繁 fallback 路径中会造成不必要的 GC 压力。

### P1-4: `_ITALIC_RE` 不跨行匹配，嵌套格式化标记可能遗漏 (telegram_format.go:36)

```go
var _ITALIC_RE = regexp.MustCompile(`\*([^*\n]+)\*`)
```

`[^*\n]+` 禁止跨行，但 Markdown 合法 italic 可以跨行（如 `*line1\nline2*`）。同理 `_STRIKE_RE` 的 `(.+?)` 跨行但 `_ITALIC_RE` 不跨行，处理不一致。

### P1-5: `sendFormattedParts` 中格式化和切分的顺序问题间接导致 split 不可靠 (telegram_text.go:411-415)

详见 P0-1。建议：将 `formatMessage` 延迟到 split 之后，分别对每个 chunk 调用（或者 split 前先计算 raw 长度，split raw text 再分别 format）。

## P2 — 中等缺陷

### P2-1: `stripANSI` 不处理冒号分隔的 CSI 序列 (main.go:41)

```go
var ansiCSI = regexp.MustCompile(`\x1b\[[0-9;?]*[a-zA-Z]`)
```

`[0-9;?]*` 不包含 `:`，因此 `\x1b[38:5:10m`（子参数分隔）不会被匹配。现代终端模拟器使用的某些 SGR 序列会泄漏。

### P2-2: `stripANSI` 不处理 ST 终止的 OSC 序列 (main.go:42)

```go
var ansiOSC = regexp.MustCompile(`\x1b\][^\x07]*\x07`)
```

只处理 BEL(`\x07`) 终止的 OSC，不处理 ST(`\x1b\\`) 终止的 OSC。虽然 `ansiBare` 会单独剥离 `\x1b]` 和 `\x1b\\`，但 OSC 中间的内容（如窗口标题文字）会泄漏到输出中。

### P2-3: `_escapeBareChars` 中 `)` 检测启发式算法脆弱 (telegram_format.go:398-426)

```go
if strings.Contains(before, "](http") || strings.Contains(before, "](") {
```

用子串匹配判断右括号是否属于 Markdown 链接，可能误判。例如文本中出现 `]` + `(` 但并非链接的情况。虽然 placeholder 系统已经保护了正确的链接，但回退路径可能错误地不转义某些独立的 `)` 字符。

### P2-4: `stripMdv2` 的 italic 正则有新行歧义 (telegram_format.go:450)

```go
cleaned = regexp.MustCompile(`_([^_
]+)_`).ReplaceAllString(cleaned, `$1`)
```

正则跨行书写，内容为 `_([^_\n]+)_`。虽意图限制不跨行防止误伤，但若 MDV2 格式产生跨行 `_italic_`（当前 `formatMessage` 不会产生，但其他路径可能），strip 会遗漏。

### P2-5: 各个 `ParseMode = "MarkdownV2"` 硬编码字符串不统一 (多处)

有的地方用 `tgbotapi.ModeMarkdownV2`（如 telegram_text.go:413），有的用字符串字面量 `"MarkdownV2"`（如 callback.go:172, handler.go:171, stream.go:518）。应统一使用标准常量，避免因版本升级导致的差异。

### P2-6: `appendChunk` 中对 chunk 调用 `isReasonixNoise` 语义不精确 (serve_backend.go:1048)

```go
if clean == "" || isReasonixNoise(clean) {
    return
}
```

`isReasonixNoise` 设计用于整行判断，但 `clean` 可能是流式 chunk（不完整的行）。部分行首匹配（如 `· ` 开头的 chunk）可能导致误判丢弃。

## P3 — 轻微缺陷 / 建议

### P3-1: `_BLOCKQUOTE_RE` 不支持连续块引用 (telegram_format.go:45)

```go
var _BLOCKQUOTE_RE = regexp.MustCompile(`(?m)^((?:\*\*)?>{1,3}) (.+)$`)
```

每行单独匹配，不支持多行块引用（>> 连续多行）。`formatMessage` 处理后，只有第一行会带 `>`。

### P3-2: `_TABLE_SEPARATOR_RE` 在无空格的表分隔行可能不匹配 (telegram_format.go:48)

```go
var _TABLE_SEPARATOR_RE = regexp.MustCompile(`^\s*\|?(\s*:?-{2,}:?\s*\|)+\s*:?-{2,}:?\s*\|?\s*$`)
```

要求分隔行每列都有 `|-...-|` 格式，但某些简化表格（如 `---|---`）可能被跳过。

### P3-3: `isReasonixNoise` 中 `reUnknownRef` 对 `"` 和 `'` 格式同样判断 (main.go:68)

`unknown (ref|tool)` 只匹配精确的这两个词，但 reasonix 可能使用其他格式。不过当前测试覆盖较全。

### P3-4: `stripThinkBlocks` 没有嵌套标签保护 (serve_backend.go:928-947)

如果 `think` 标签内又包含相同的开放标签，最内层的优先闭合。但在实际场景中可能性较低。

### P3-5: `telegram_format.go` 行 16-18 的注释误导

```go
// _MDV2_ESCAPE_RE_RE is the regex that matches every character MarkdownV2
// requires to be backslash-escaped outside code spans.
var _MDV2_ESCAPE_RE_RE = regexp.MustCompile(`([_*\[\]()~` + "`" + `>#+\-=|{}.!\\])`)
```

注释和变量名暗示这是主要转义正则，实际并不被使用。应删除或替换为引用 `_MDV2_ESCAPE_RE`。
