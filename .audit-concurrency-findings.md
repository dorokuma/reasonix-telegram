# Deep Investigation: Message Stuck → Backlog → Simultaneous Release → Duplicates

## Overview

This report documents the deep investigation into race conditions and blocking paths that could cause the "messages stuck → backlog → simultaneous release → duplicate responses" pattern in `reasonix-telegram`.

---

## 1. `/stop` Command Implementation

**Files & lines**: `handler.go:310-324`, `stream.go:23-77`, `stream.go:852-882`

### What `/stop` does:
```go
// handler.go:310-324
case text == "/stop" || text == "/cancel":
    s := a.getOrCreateSession(m.Chat.ID)
    s.mu.Lock()
    t := s.task
    s.mu.Unlock()
    if t == nil {
        a.reply(m.Chat.ID, "当前没有进行中的回复")
        return
    }
    t.cancel()              // just calls context.CancelFunc
    a.reply(m.Chat.ID, "🛑 已发送中止信号")
    return
```

`t.cancel()` calls the `context.CancelFunc` created in `runTask` (`stream.go:33`). This cancels the context associated with the running task.

### Why typing continues after `/stop`:

The `stopTyping` function is set up at `stream.go:66-67`:
```go
stopTyping := a.beginTyping(chatID)
defer stopTyping()
```

`beginTyping` (handler.go:571-589) creates a **separate context** (`context.WithCancel(context.Background())`) and spawns a goroutine that refreshes the Telegram "typing…" indicator every 4 seconds. The returned `cancel` function stops this goroutine.

However, `stopTyping()` is **deferred** and only runs when `runTask` **returns**. After context cancellation, `runTask` enters a select at `stream.go:852-862` that waits up to **8 seconds** for the pusher goroutine to finish:

```go
select {
case <-pusherDone:
case <-ctx.Done():
    select {
    case <-pusherDone:
    case <-time.After(8 * time.Second):  // <-- blocks up to 8s
        if procErr == nil {
            procErr = ctx.Err()
        }
    }
}
```

During these 8 seconds, `runTask` hasn't returned yet, so the deferred `stopTyping()` hasn't been called, and the typing indicator keeps refreshing.

Additionally, the `runServeTurn` function (`serve_backend.go:830-839`) also waits up to 8 seconds after cancellation:
```go
case <-ctx.Done():
    _ = a.postJSON(port, "/cancel", map[string]any{})
    select {
    case tr := <-eventsDone:
        return tr.err
    case <-time.After(8 * time.Second):
        return ctx.Err()
    }
```

**Total delay**: Up to ~16 seconds (8s in `runServeTurn` + 8s in `runTask`) before typing stops.

### Does `/stop` actually cancel the running task?

**Partially.** The context cancellation propagates to:
- ✅ The SSE connection (created with `http.NewRequestWithContext(ctx, ...)` at serve_backend.go:853)
- ✅ A POST `/cancel` is sent to the serve process (serve_backend.go:833)
- ❌ But the POST `/submit` request (serve_backend.go:824) uses `postJSON` which uses `context.Background()` — not cancellable

## 2. HTTP Request Timeout Mechanism

**Files & lines**: `serve_backend.go:106-111`, `serve_backend.go:783-806`, `serve_backend.go:848-860`

### Two HTTP clients:

```go
// serve_backend.go:106-111
var localHTTPClient = &http.Client{Timeout: 10 * time.Second}   // for POST /submit, /cancel, /status
var sseClient = &http.Client{}                                    // NO timeout for SSE stream
```

### POST /v1/chat/completions (actually POST /submit):

`runServeTurn` (`serve_backend.go:824`) calls `a.postJSON(port, "/submit", ...)`.

**`postJSON` implementation** (`serve_backend.go:783-806`):
```go
func (a *App) postJSON(port int, path string, body any) error {
    // ...
    req, err := http.NewRequestWithContext(context.Background(), ...)  // <-- uses Background()!
    // ...
    resp, err := localHTTPClient.Do(req)    // 10s timeout
    // ...
}
```

**Critical flaw**: `postJSON` uses `context.Background()` instead of the cancellable task context. This means:
1. If the serve back-end hangs, `localHTTPClient`'s 10-second timeout will fire — ✅ protected
2. But if the task context is cancelled while POST /submit is in flight, the HTTP request is NOT cancelled — ⚠️ it runs to completion (10s timeout or success)
3. POST /cancel to serve uses the same background context — this is actually ✅ correct (we want cancel to always reach serve even if task context is done)

### SSE connection timeout:

The SSE stream (`connectAndConsumeSSE` at `serve_backend.go:848-860`):
- Uses `sseClient` (no overall timeout) — by design for long-lived streams
- Request created with `http.NewRequestWithContext(ctx, ...)` — ✅ context cancellation propagates
- Has an **idle watchdog** (serve_backend.go:897-916) that closes the connection after **5 minutes** of inactivity
- Retries on transient failures: 3 retries with backoff (500ms, 5s, 30s)

**If serve back-end hangs**: The SSE stream sits idle for 5 minutes before the watchdog fires. During this time, the handler goroutine is "stuck" holding a slot in `handlerSem`.

## 3. `ensureServe` Lock Mechanism

**Files & lines**: `serve_backend.go:560-574`, `serve_backend.go:240-252`, `serve_backend.go:324-558`

### Implementation:

```go
func (a *App) ensureServe(chatID int64) error {
    s := a.getOrCreateSession(chatID)
    if a.serveRunning(s) {
        s.mu.Lock()
        port := s.servePort
        s.mu.Unlock()
        if port > 0 {
            return a.waitServeReady(port, 10*time.Second)
        }
        return nil
    }
    return a.startServe(chatID, true)
}
```

`serveRunning` (`serve_backend.go:240-252`):
```go
func (a *App) serveRunning(s *session) bool {
    s.mu.Lock()
    cmd := s.serveCmd
    _ = s.servePort
    s.mu.Unlock()
    if cmd != nil && cmd.Process != nil && cmd.ProcessState == nil {
        return true
    }
    return false
}
```

### Lock analysis:
1. `ensureServe` calls `getOrCreateSession` which locks `a.sessMu` — the global session map lock
2. `serveRunning` locks `s.mu` (per-session lock) to read `s.serveCmd`
3. `startServe` checks `s.serveCmd` under lock at line 330 — mitigates TOCTOU between `serveRunning` returning false and `startServe`

### Potential issues:

**P1 — TOCTOU race** (`serve_backend.go:562-573`):
- `serveRunning(s)` returns false (under `s.mu` lock, then unlocks)
- Between the unlock and `startServe` call, another goroutine could call `startServe` first
- **Mitigation**: `startServe` re-checks `s.serveCmd` under `s.mu` lock at line 330, returns nil if already running

**P1 — `ensureServe` called without per-chat serialization**:
- Two concurrent `runTask` calls for the same chat can both call `ensureServe`
- The first one succeeds (serve already running or starts it)
- The second one also succeeds (serve is now running)
- No harm since it's idempotent, but wasteful

**P0 — If serve process fails to start** (`serve_backend.go:536-539`):
- `startServe` returns error
- `runTask` handles it at `stream.go:54-64`:
  ```go
  if err := a.ensureServe(chatID); err != nil {
      s.mu.Lock()
      if s.task == thisTask { s.task = nil }
      s.mu.Unlock()
      cancel()
      a.reply(chatID, fmt.Sprintf("Reasonix 服务启动失败: %v", err))
      return
  }
  ```
  ✅ Properly cleans up

## 4. `handlerSem` Semaphore Mechanism

**Files & lines**: `main.go:368`, `main.go:477`, `main.go:568-589`

### Creation:
```go
// main.go:477
handlerSem: make(chan struct{}, 100),
```

### Usage in main loop:
```go
// main.go:568-589
for {
    select {
    case <-ctx.Done():
        // shutdown
    case upd, ok := <-updates:
        if !ok { return }
        if upd.Message != nil {
            app.msgWg.Add(1)
            app.handlerSem <- struct{}{}   // <-- BLOCKS when 100 in flight
            go func() {
                defer func() { <-app.handlerSem }()
                app.handleMessage(upd.Message)
            }()
        }
        // same for CallbackQuery
    }
}
```

### Analysis:

| Property | Value |
|----------|-------|
| Capacity | 100 |
| What it limits | Concurrent `handleMessage`/`handleCallbackQuery` goroutines across ALL chats |
| Blocking behavior | When full (100), the main loop **blocks** — no new updates are read from the `updates` channel |
| What happens to Telegram updates during block | They buffer in the `updates` channel (Telegram long-poll keeps receiving, the bot library puts them in the channel) |

**P0 — Semaphore full causes update pileup**:
- If 100 handler goroutines are stuck (e.g., waiting on SSE streams with 5-min idle timeout, or slow HTTP requests), the main loop blocks on `handlerSem <- struct{}{}`
- Telegram updates keep arriving and buffering in the `updates` channel
- When handlers eventually finish (SSE timeout), the semaphore releases slots
- All buffered updates are processed nearly simultaneously
- This causes a "burst" of `handleMessage` calls for potentially the same chat

**P1 — Per-chat rate limiter mitigates but doesn't eliminate the burst**:
- `handler.go:34-43`: Drops messages from the same chat if <3s apart
- But if 10 messages for chat X were buffered, they'd all arrive at the main loop with fresh `time.Now()` timestamps (they were in the channel, not in the main loop)
- The first message passes; the rest might be within 3s of each other and get dropped
- **BUT**: If there's a gap (e.g., messages spaced 5s apart originally), they could all pass the rate limiter

## 5. Task Cancellation Mechanism

**Files & lines**: `main.go:358-360` (`runningTask`), `stream.go:23-50` (task lifecycle), `stream.go:854-882` (cancel handling), `serve_backend.go:829-839` (serve turn cancel)

### The `runningTask` struct:
```go
type runningTask struct {
    cancel context.CancelFunc
}
```
Only a `CancelFunc` — no mutex, no done channel, no state.

### Cancel flow:
1. `/stop` or pre-emption calls `s.task.cancel()` (the context.CancelFunc)
2. `runServeTurn`'s select detects `ctx.Done()` at `serve_backend.go:832`
3. Sends POST `/cancel` to serve process (`serve_backend.go:833`)
4. Waits up to 8s for SSE events to drain (`serve_backend.go:834-839`)
5. Returns `ctx.Err()`
6. `runTask`'s select at `stream.go:854` detects context cancellation
7. Waits up to 8s for pusher goroutine (`stream.go:855-862`)
8. Sends error message to chat ("已中止" or "超时")
9. Deferred cleanup runs: `stopTyping()`, cancel(), task=nil

### Critical path: Does context cancellation actually stop the HTTP request?

**SSE connection** (`serve_backend.go:853`):
```go
req, err := http.NewRequestWithContext(ctx, http.MethodGet, eventsURL, nil)
```
✅ YES — context cancellation cancels the SSE HTTP request. The Go HTTP client closes the connection when the context is cancelled.

**POST /submit** (`serve_backend.go:788` inside `postJSON`):
```go
req, err := http.NewRequestWithContext(context.Background(), ...)
```
❌ NO — uses `context.Background()`, not the task context. The POST /submit runs to completion regardless of cancellation.

**POST /cancel** (`serve_backend.go:788` inside `postJSON`):
Same `context.Background()` — ✅ intentionally, so the cancel signal always reaches serve.

### ⚠️ Race condition in task cleanup:

`runTask` line 69-78:
```go
defer func() {
    s.mu.Lock()
    if s.task == thisTask {
        s.task = nil
        s.wakePusher = nil
    }
    s.mu.Unlock()
    cancel()
}()
```

This only clears `s.task` if it's still `thisTask`. If a newer task has already replaced it (via pre-emption), the old task doesn't clear the reference. This is correct — the new task's cleanup will handle it.

**But**: There's a window between the task being pre-empted (line 41-50 sets new task, oldCancel fires) and the old task's `runServeTurn` returning. During this window, `s.task` points to the NEW task, but the OLD task's goroutine is still running (draining SSE events, waiting for 8s timeout). If the new task calls `ensureServe` during this window, it proceeds normally because `ensureServe` only checks `serveRunning` (the serve process), not `s.task`.

## 7. Per-Chat Request Queue Analysis

**Finding: No per-chat queue exists.**

Messages from the same chat are processed in parallel goroutines with no ordering guarantee:

1. `main.go:576-579`: Each message spawns a new `go func()` for `handleMessage`
2. `handler.go:34-43`: Per-chat rate limiter drops messages <3s apart
3. Messages >3s apart from the same chat **run concurrently**
4. Each calls `runTask` independently
5. `runTask` uses pre-emption (cancel old task, set new task) to handle concurrent requests
6. But the pre-emption takes time — the old task's HTTP requests and SSE connections are still active during the transition

### The "Backlog Release" Chain:

```
1. 100 SSE connections active → handlerSem full
2. Main loop BLOCKS on handlerSem <- struct{}{}
3. Telegram updates buffer in `updates` channel (unbounded)
4. SSE connections timeout (5-min idle watchdog)
5. Handler goroutines finish → release semaphore slots
6. Main loop drains buffered updates rapidly
7. Each chat may get multiple handleMessage calls in quick succession
8. Each handleMessage → runTask → pre-empts previous
9. Serve process receives rapid /cancel + /submit sequences
10. If the serve process doesn't fully cancel before new /submit arrives,
    old SSE connection MAY still deliver events (⚡DUPLICATE⚡)
```

## 8. Root Cause Summary: P0 Findings

### 🔴 P0-1: `postJSON` uses `context.Background()` (serve_backend.go:788)

```go
req, err := http.NewRequestWithContext(context.Background(), ...)
```

Task context cancellation does NOT cancel in-flight HTTP requests to serve. The POST /submit can complete even after the task was cancelled. This is partially mitigated by the 10s client timeout.

### 🔴 P0-2: No per-chat serialization for handleMessage

Messages >3s apart from the same chat run in concurrent goroutines. The pre-emption mechanism mitigates but doesn't eliminate race conditions during the transition window.

### 🔴 P0-3: handlerSem full causes unbounded update buffering

When 100 handlers are stuck (SSE idle timeout is 5 minutes!), the main loop blocks, updates buffer, then flood in when slots free up.

### 🔴 P0-4: Typing indicator continues up to 16s after `/stop`

`stopTyping` is deferred and runs only after `runTask` returns, which waits up to 8s (in runServeTurn) + 8s (in runTask) = 16s total.

### 🟡 P1-1: `/stop` doesn't kill the serve process

It only cancels the Go context and sends POST /cancel. If the serve process is in a bad state (blocked on LLM API), the cancel signal might not propagate quickly enough.

### 🟡 P1-2: SSE idle watchdog at 5 minutes is excessive

A handler goroutine stuck on an SSE stream holds a semaphore slot for up to 5 minutes before the watchdog fires. Reducing this could prevent backlog.

## 9. Recommended Fixes

### 1. Use cancellable context in `postJSON` (serve_backend.go:788)
```go
// Instead of context.Background(), accept and use a context:
func (a *App) postJSON(ctx context.Context, port int, path string, body any) error {
    req, err := http.NewRequestWithContext(ctx, ...)
```

### 2. Add per-chat mutex for sequential message processing
Add a per-chat mutex (e.g., `chatMu sync.Map`) in handleMessage to serialize messages from the same chat:
```go
mu, _ := a.chatMu.LoadOrStore(chatID, &sync.Mutex{})
mu.Lock()
defer mu.Unlock()
```

### 3. Reduce SSE idle watchdog timeout (serve_backend.go:908)
Change `300` to `120` (2 minutes) to reduce the stuck-handler window.

### 4. Make typing stop synchronous on `/stop`
Call `stopTyping()` directly in the `/stop` handler instead of relying on the deferred call in `runTask`.

### 5. Add a request queue per chat
Instead of dropping messages or processing concurrently, queue them per-chat and process sequentially.
