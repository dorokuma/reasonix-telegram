## 凭证与密钥管理安全审计 — 证据收集

### 1. 密钥加载路径追踪

**TG_BOT_TOKEN**: 
- 来源: os.Getenv("TG_BOT_TOKEN") (main.go:254)
- 去向: Config.BotToken → secrets list → tokenRedactingTransport.token → 最终被 os.Unsetenv (main.go:420)
- 传递: 通过 HTTP Authorization header 发送到 api.telegram.org (tgbotapi 内部)

**DEEPSEEK_API_KEY**:
- 来源: loadEnvFile("/etc/reasonix-api.env", "DEEPSEEK_API_KEY") (main.go:260)
- 去向: Config.DeepSeekKey → secrets list → reasonixEnv() 加入 child env (serve_backend.go:215) 同时写入临时文件 (serve_backend.go:400-406)
- 双通道传递: 同时通过环境变量 `DEEPSEEK_API_KEY` 和 `DEEPSEEK_API_KEY_FILE` 传递给子进程

**CF_TOKEN**:
- 来源: os.Getenv("CF_TOKEN") (main.go:280)
- 去向: 仅用于收集到 secrets 列表用于日志脱敏，然后被 os.Unsetenv (main.go:421)
- 无其他用途

### 2. sessioncrypt.go 分析

- 文件定义: DecryptWithAAD, Decrypt, IsEncrypted, loadOrCreateKey, KeyPath
- 缺失: **没有 Encrypt 函数**
- 密钥存储: `~/.config/reasonix/.session-key`，目录 0700，文件 0600
- AES-GCM 实现: 使用 crypto/aes + crypto/cipher.NewGCM，nonce 从 payload 读取 (偏移 magicLen)，验证标签由 GCM 自动验证
- 调用链: persist.go:sessionStats() 调用 IsEncrypted → Decrypt 读取 session JSONL

### 3. tokenRedactingTransport 分析

- 位置: telegram_bridge.go:123-137
- 行为: 仅从 `err.Error()` 中替换 token 字符串，不处理 URL、请求体、响应体
- tryIP() (telegram_network.go:184) 在 HTTP Transport 层之外创建了新连接，有自己的内联 token 脱敏

### 4. reasonixEnv() 白名单分析

- 安全前缀列表 (serve_backend.go:179-186): HOME, USER, LOGNAME, SHELL, PATH, LANG, LC_, LANGUAGE, TZ, HTTP_PROXY, HTTPS_PROXY, NO_PROXY, http_proxy, https_proxy, no_proxy, EDITOR, VISUAL, PAGER, XDG_CACHE_HOME, XDG_CONFIG_HOME, XDG_DATA_HOME, XDG_STATE_HOME
- 只添加 DEEPSEEK_API_KEY 到子进程环境
- 风险: HTTP_PROXY 变量可能被用于代理劫持

### 5. Unsetenv + /proc/PID/environ

- main.go:419-421: 在 loadConfig() 之后调用 os.Unsetenv 清除 TG_BOT_TOKEN, DEEPSEEK_API_KEY, CF_TOKEN
- 未清除: TELEGRAM_FALLBACK_IPS, 和其他从父进程继承的 env vars
- 实测: 当前进程的 /proc/PID/environ 已没有 TG_BOT_TOKEN, DEEPSEEK_API_KEY, CF_TOKEN（确认有效）

### 6. /etc/reasonix-api.env 权限

- 文件权限: 600 (owner read-write only) ✓
- loadEnvFile 有权限检查逻辑 (main.go:224-228)
- 路径硬编码: 不可配置，不支持多 API key 轮换
