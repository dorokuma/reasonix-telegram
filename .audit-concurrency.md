# 并发安全审计报告：reasonix-telegram

审计日期: 2026-06-18
审计范围: 所有 goroutine、锁、channel、WaitGroup 使用
审计方法: 静态代码分析（基于 grep 和代码阅读）

## 总体评分: 7/10

架构设计良好，大部分并发模式使用正确。主要问题集中在：goroutine 退出路径不完整、锁保护范围不足、以及潜在的 goroutine 泄漏风险。

---

## P0 — 必须修复

### P0.1 goroutine 泄漏：beginTyping 的 goroutine 无退出保证

**文件**: `handler.go:498-509`  
**问题**: `beginTyping` 启动的 goroutine 监听 `ctx.Done()` 后返回。但如果 `runTask` 函数返回时忘记调用返回的 `cancel()` 函数，这个 goroutine 会永远运行。  
**现状**: 唯一的调用点在 `stream.go:52-53`，使用了 `defer stopTyping()`，所以实际不会泄漏。但此函数作为公共 API 暴露，未来可能被误用。  
**建议**: 无修复必要，但建议在函数注释中强调必须 defer 返回值。

### P0.2 锁顺序反转风险：s.mu → a.state.upsert (st.mu)

**文件**: `stream.go:646-689`  
**问题**: `onUsage` 回调中，`s.mu` 被持有（646行），然后在 unlock 后调用 `a.state.upsert(rec)`（689行）。经确认，`s.mu` 确实在 upsert 前已释放 (687行 unlock)，**当前没有死锁**。但这是一个脆弱的设计——如果未来有人修改代码在 `s.mu` 持有期间调用 `state.*` 方法，会导致死锁（因为 `startServe` 可能在 `st.mu` 下调用 `s.mu`）。  
**建议**: 保持当前顺序不变，添加注释警告不要嵌套。

### P0.3 并发 map 写：clarifyState.answers 在多个 goroutine 中并发写入

**文件**: `handler.go:148` 和 `callback.go:576`  
**问题**: `pc.answers[pc.questionID]` 在 `handleMessage` goroutine 和 `handleCallbackQuery` goroutine 中都被修改。两者都先通过 `s.mu.Lock()` 访问 session，所以 pc 的访问是受锁保护的。但答案 map 的写入在不同 goroutine 中通过同一个锁保护，只要锁使用正确就没有问题。  
**验证**: 两处代码都在 `s.mu.Lock()`/`s.mu.Unlock()` 块内访问 `pc.answers`，所以并发安全成立。  
**建议**: 无需修复。

### P0.4 main 循环中未等待 msgWg 就 return

**文件**: `main.go:493-495`  
**问题**: 当 `updates` 通道关闭时（`if !ok { return }`），main 函数立即返回而不等待 `msgWg`。这会导致正在执行的 `handleMessage` 和 `handleCallbackQuery` goroutine 被遗弃。  
**并发场景**: 如果 Telegram bot API 连接断开导致 updates 通道关闭，但此时仍有 message handler 在执行（如正在下载大文件），这些 goroutine 会在后台泄漏。  
**建议**: 在 `return` 前添加 `app.msgWg.Wait()`。

## P1 — 建议修复

### P1.1 health check goroutine 无法被停止

**文件**: `serve_backend.go:491`  
**问题**: `startServeHealthCheck()` 启动的 goroutine 使用 `ticker.C` 无限循环，没有退出机制。进程退出时它会被杀死，但如果需要优雅关闭，这个 goroutine 无法被取消。  
**并发场景**: 进程收到 SIGTERM 时，该 goroutine 可能在 `a.sessMu.Lock()` 上阻塞，而 `stopAllServes()` 也在获取 `sessMu`。这不构成死锁（因为 ticker 循环最终会释放锁），但可能导致优雅关闭延迟。  
**建议**: 添加 `ctx` 参数，让 goroutine 监听取消信号。

### P1.2 `connectAndConsumeSSE` 中 watchdog goroutine 竞争关闭 body

**文件**: `serve_backend.go:756-762` 和 `778-794`  
**问题**: 有两个 goroutine 可以关闭 `resp.Body`：
1. 756行的 goroutine 在 ctx.Done() 时关闭 body
2. 778行的 watchdog goroutine 在 idle 超时时关闭 body

如果两者同时触发，可能导致 `resp.Body.Close()` 被调用两次。`http.Response.Body` 的 `Close` 通常是幂等的，但取决于具体实现。  
**建议**: 使用 `sync.Once` 确保 body 只关闭一次。

### P1.3 `sendTextParts` 中 `recordSentText` 与 `saveSentTextCache` 的锁嵌套

**文件**: `utils.go:233-249` 和 `266-297`  
**问题**: `recordSentText` 在 `sync.Map.Range` 循环中删除条目（安全），然后调用 `saveSentTextCache`。`saveSentTextCache` 遍历整个 `sync.Map` 进行序列化，同时获取 `sentTextCacheMu`。如果多个 goroutine 同时调用 `recordSentText`，虽然 `sync.Map` 安全，但 `sentTextCacheMu` 会串行化磁盘写入，可能导致频繁的 JSON 序列化和磁盘 I/O。  
**建议**: 考虑减少持久化频率（如每 N 条记录写一次，或使用 debounce timer）。

### P1.4 `seenMsgs` 和 `rateLimits` 无限增长

**文件**: `handler.go:18` 和 `main.go:374`  
**问题**: `seenMsgs` sync.Map 从未清理过期条目；`rateLimits` sync.Map 也从未清理。对于长期运行的 bridge，可能导致内存增长。  
**并发场景**: 高流量 chat 下，message ID 不断累加，sync.Map 持续增长。  
**建议**: 添加定期清理或 LRU 淘汰。

## P2 — 注意

### P2.1 `flushNow` 通道容量为 1 可能丢信号

**文件**: `stream.go:84`  
**问题**: `flushNow` 是容量为 1 的缓冲通道。`signalFlush` 使用非阻塞发送。如果 `runServeTurn` 的 SSE 回调中的 `onComplete` 和 `onUsage` 几乎同时触发，后者的 flush 信号可能被丢弃。  
**影响**: 最终消息仍会通过 `endStream()` 发送（因为 `finished` 通道和最后的 `buf` 检查），所以不会丢失内容。  
**建议**: 保持现状，设计上可接受。

### P2.2 `stopServe` goroutine 在极端情况下泄漏

**文件**: `serve_backend.go:270-273`  
**问题**: 如果 `cmd.Process.Wait()` 永远不返回（如进程已经是 zombie），goroutine 永久泄漏。但这种情况极罕见，因为后续有 SIGKILL 和超时处理。  
**建议**: 无需修复，但可考虑将 Wait 也纳入超时管理。

### P2.3 `msgWg.Add(1)` 在 goroutine 内部而不是外部

**文件**: `handler.go:21` 和 `callback.go:394`  
**问题**: `msgWg.Add(1)` 在 goroutine 内部调用，`main.go:498` 是 `go app.handleMessage(upd.Message)`。如果 goroutine 调度延迟，可能在 `msgWg.Add(1)` 执行前 main 就已经到达 `msgWg.Wait()` 并返回。虽然 Go 的调度器在 `go` 语句后通常很快执行被调函数，但并发模型上这是一个常见的竞态模式。  
**建议**: 在 `main.go:498-501` 的 `go` 语句前执行 `msgWg.Add(1)`。

### P2.4 session.serveCmd 在多 goroutine 间存在数据竞争窗口

**文件**: `serve_backend.go:232-235` 和 `serve_backend.go:438-443`  
**问题**: `startServe` 设置 `s.serveCmd = cmd` 后启动 `cmd.Wait()` goroutine。如果 `stopServe` 在 `startServe` 设置 `serveCmd` 之后但在 Wait goroutine 获取锁之前被调用，`stopServe` 会读取刚设置的 `cmd`（没问题），但 Wait goroutine 可能看到 `s.serveCmd` 已被置为 nil（没问题，因为条件 `s.serveCmd == cmd` 不成立）。这个 race 窗口很小且不导致不正确行为。  
**建议**: 保持现状。
