### 🔴 P0 — restart.go:254: restart watchdog goroutine leaks forever without exit path

```go
func (a *App) startRestartWatchdog() {
    restartWatch.Do(func() {
        go func() {
            t := time.NewTicker(15 * time.Second)     // ← never Stopped
            for range t.C {                           // ← iterates forever
                ...
            }
        }()
    })
}
```

- **Risk**: Every program start spawns a goroutine that `range`s over a `time.Ticker` channel. The ticker is **never Stopped**, and the loop has **no break/return condition** — it runs until process exit. While this doesn't cause a leak per se (the goroutine is alive until the process dies), it's a design smell: on shutdown (SIGTERM path in main), the watchdog is left running, holding a ticker that's never cleaned up. The goroutine continues to run during shutdown, which could race with `a.restarting` / `a.restartStarted` writes (though those are under `restartMu`).
- **Fix**: Add a shutdown channel parameter or derive from the app's `healthCheckStop`/shutdown context so the goroutine exits gracefully during shutdown. `ticker.Stop()` should also be deferred.

### 🔴 P0 — restart.go:238: goroutine spawned during /restart does not respect shutdown signal

```go
go func() {
    a.cancelAllTasks()
    a.waitTasksDone(30 * time.Second)   // ← blocks up to 30s
    a.stopAllServes()
    time.Sleep(500 * time.Millisecond)
    triggerSystemRestart()              // ← os.Exit(1), never returns
}()
```

- **Risk**: If a SIGTERM arrives while this goroutine is waiting, the main shutdown sequence (main.go:511-527) also calls `a.cancelAllTasks()` + `a.waitTasksDone(5*time.Second)` + `a.stopAllServes()`. These run **concurrently** with this goroutine's calls. The double-kill on child processes (from both paths racing) is mostly harmless because `stopServe` uses `sync.Once`-like idempotency (sets `s.serveCmd = nil`), but the race window in `stopSessionServe` (serve_backend.go:254-255: `cmd := s.serveCmd; s.serveCmd = nil`) could allow SIGTERM to be sent to a `nil`-ed cmd.Process. Also, two concurrent `stopAllServes()` loops mean some serves might get killed twice.
- **Fix**: `gracefulServiceRestart` should listen to the same shutdown context. Either block shutdown from proceeding during a manual restart (set a flag), or cancel the restart goroutine when SIGTERM arrives.

### P1 — main.go:539-544: handleMessage/handleCallbackQuery goroutines spawned without limiting

```go
go app.handleMessage(upd.Message)          // msgWg.Add(1) / Done() in callee
go app.handleCallbackQuery(upd.CallbackQuery)
```

- **Risk**: No upper bound on concurrent goroutines. Under Telegram update flood (e.g. many users sending messages simultaneously), the number of goroutines is unbounded — each does HTTP calls, starts serve processes, etc. The `rateLimits` sync.Map provides per-chat rate limiting at the handler entry, but across unrelated chats this is still unbounded.
- **Fix**: Use a worker pool pattern or a semaphore (e.g. `chan struct{}` with capacity 100) that blocks the event loop when too many handlers are in flight, preventing OOM.

## 2. RACE CONDITIONS

### 2.1 session struct field access

The `session` struct (main.go:326-348) uses `s.mu sync.Mutex` to protect its fields. Most access is properly locked, but several patterns are concerning:

### 🔴 P0 — serve_backend.go:507: TOCTOU race on serveCmd / ProcessState

```go
// serve_backend.go:507 (startServeHealthCheck)
alive := cmd != nil && cmd.Process != nil && cmd.ProcessState == nil
```

This checks `cmd.ProcessState` **outside the lock** after reading `cmd` under the lock. The `cmd` value itself is a pointer taken under the lock, but between reading `cmd` and checking `cmd.ProcessState`, another goroutine (e.g., `cmd.Wait()` goroutine at line 431-441) could mutate `cmd.ProcessState` by calling `cmd.Wait()`. However, `ProcessState` is set by Go's runtime when Wait returns — if the Wait goroutine hasn't run yet, ProcessState could still be nil. The worst case is a false positive (thinking the process is alive when it just exited). Low practical risk but a data race in theory.

### P2 — serve_backend.go:507-508: reading serveCmd and servePort in separate lock windows

```go
s.mu.Lock()
cmd := s.serveCmd
port := s.servePort
s.mu.Unlock()
alive := cmd != nil && cmd.Process != nil && cmd.ProcessState == nil
listening := port > 0 && a.waitServeReady(port, 2*time.Second) == nil
```

Between the unlock and the `listening` check, another goroutine could have changed the port. The window is tiny and `waitServeReady` with a wrong port would just fail quickly — low risk.

### P2 — serve_backend.go:517-519 / 530-531 / 537-548: Health check restart races with stopServe

In the health check restart loop (serve_backend.go:514-550), `s.serveRestartCount`, `s.serveLastRestart` are accessed with lock-then-unlock patterns. If `stopServe` or `stopAllServes` runs concurrently (from shutdown or a manual /restart), the health check could restart a serve process that was intentionally stopped. The `s.serveCmd = nil` pattern in `stopSessionServe` mitigates this — after stop, any restart in the health check would call `startServe` which sees `serveCmd == nil` and starts fresh. The window between the check and the restart is small, but exists.

### P2 — handler.go:34-39: rateLimits LoadOrStore + Store race (benign but intentional)

The rate limit uses `LoadOrStore` then later `Store` with a new timestamp. Between `LoadOrStore` and `Store`, another message could slide through. This is a deliberate design choice (permissive rate limiting) and not a bug, but worth noting.

### 2.2 Clarify/Approval state races

### P2 — handler.go:134-212: pendingClarify mutation across lock/unlock cycles

In `handleMessage`, the clarify state is accessed under `s.mu.Lock()`, then the lock is dropped for message sending, then re-acquired. The protect pattern is:

```go
s.mu.Lock()
pc := s.pendingClarify
// ... check pc fields, pc.answers[pc.questionID] = ...
nextIdx := pc.qIndex + 1
// ... more mutation ...
s.mu.Unlock()

// ... send message (no lock) ...

s.mu.Lock()
s.pendingClarify.messageID = sent.MessageID
s.mu.Unlock()
```

Between the two lock windows, a callback query goroutine (handleCallbackQuery) could also mutate `pendingClarify` under the same lock. This is a real race: if a user presses a callback button **while** a text message is being processed concurrently (both arrive within the same update batch), both goroutines could read and write `pc.answers`, `pc.qIndex`, etc. The `seenMsgs` dedup prevents same-message duplication but doesn't prevent a callback arriving concurrently with a text message. **Two goroutines could both advance `qIndex` and overwrite each other's answers.**

### P3 — callback.go:552-599: Similar race in callback handler

Same pattern — `s.mu.Lock()`, read `pc`, unlock, send message, lock again. Another clarify mutation between those windows could corrupt state.

### 2.3 Media group operations

### P2 — media_group.go:42-73: enqueueMediaGroup race with flushMediaGroup

`enqueueMediaGroup` holds `mediaGroupsMu` and starts a timer (`time.AfterFunc`). The timer fires asynchronously and calls `flushMediaGroup`, which also acquires `mediaGroupsMu`. The timer fires **after** `enqueueMediaGroup` releases the lock. But consider:

1. Goroutine A: enqueueMediaGroup for group X, starts timer.
2. Timer fires on goroutine B: flushMediaGroup acquires lock, deletes batch, releases lock.
3. Goroutine C: enqueueMediaGroup for group X (same group ID, e.g., user sends more photos with same group ID). Creates new batch.
4. Timer from step 1 fires (it was the old timer? No, `batch.timer.Stop()` was called in step 1... wait, let me re-read.)

Looking at enqueueMediaGroup:
```go

if !exists {
    batch = &mediaGroupBatch{...}
    a.mediaGroups[chatID][groupID] = batch
}
if batch.timer != nil {
    batch.timer.Stop()            // Stop old timer
}
batch.photos = append(...)
batch.timer = time.AfterFunc(mediaBatchDelay, func() {
    a.flushMediaGroup(chatID, groupID)

```

The `timer.Stop()` returns false if the timer already fired, which is fine (the old batch was flushed). The new timer fires 800ms after the last photo is added. If photos trickle in slowly (>800ms apart), each photo flushes as a single-photo group, which is the intended behavior. **No race here** — the lock + timer.Stop() pattern is correct. ✅

### 2.4 sentTextCache operations

### P1 — utils.go:233-266: sentTextCache eviction race with concurrent store

```go
func (a *App) recordSentText(msgID int, text string) {
    count := 0
    a.sentTextCache.Range(func(_, _ any) bool { count++; return count < 500 })
    if count >= 500 {
        n := 0
        a.sentTextCache.Range(func(k, _ any) bool {
            if n < 250 {
                a.sentTextCache.Delete(k)
                n++
            }
            return true
        })
    }
    a.sentTextCache.Store(msgID, text)
```

- **Race 1**: `Range` + `Delete` is not atomic. Between counting and evicting, other goroutines (handleMessage, callback handlers) can add new entries via `Store`. The eviction target selection is arbitrary (sync.Map iteration order is random), so the cache could still exceed 500 entries. This is a low-precision cache, not correctness-critical, so it's acceptable.
- **Race 2**: `sync.Map` operations from multiple goroutines are fine individually (they're designed for concurrent use), but the `Range` → `Delete` sequence means the "evict 250" logic is approximate at best. The same entry could be deleted twice and another entry never deleted.
- **Fix**: Use a `sync.Mutex` + `map[int]string` for accurate eviction, or accept the approximation (it's a text cache for reply quoting — imprecision is harmless). **Mark as P3.**

### 2.6 stickerCache concurrency

### P1 — sticker_cache.go:72-107: stickerMu lock wraps load + save, but the cache map is loaded from disk every time

```go
a.stickerMu.Lock()
defer a.stickerMu.Unlock()
cache := a.loadStickerCache()  // reads from disk
// ... modify cache ...
a.saveStickerCache(cache)       // writes to disk
```

- **Issue**: `loadStickerCache()` reads from disk and `saveStickerCache()` writes to disk, but the in-memory `cache` map is local — it's not shared between goroutines. However, if two messages arrive simultaneously for the same chat (one photo, one sticker), two goroutines could call `handleSticker` concurrently. The `stickerMu` lock prevents interleaving, but each call does a full disk read → modify → write. This is correct but inefficient (read/write disk on every sticker).
- **Perf concern only**, not a correctness bug, but worth noting as it bypasses the in-memory cache on every call because `loadStickerCache()` is called inside the locked method, not stored as a persistent map. Actually, looking closer, `loadStickerCache()` creates a new map each time. The cache is only persisted per-sticker, not per-call. So each call re-reads the whole file from disk to check if a sticker is cached. For the second concurrent call, the first call's write might not be on disk yet (if the disk write is slow). But since `stickerMu` serializes access, this is okay — just slower than holding the map in memory.
- **Fix**: Keep the sticker cache in memory (e.g., `a.stickerCache map[string]StickerCacheEntry` protected by `stickerMu`), and only write to disk on mutations. Load once at startup.

### 3.1 Channel close safety

### P2 — stream.go:84-86: flushNow, pushWake, newSegment are buffered channels with capacity 1

```go

pushWake   = make(chan struct{}, 1)
```

These are used with non-blocking sends (select with default) which is correct:

The receiver `pusherDone` loop (stream.go:788-824) listens on `<-finished`, which fires when closed. No sends to `finished` — it's a signal-only channel.

### 3.2 Deadlock risks

### P2 — stream.go:827-837: select between pusherDone and ctx.Done with nested select

```go
select {
case <-pusherDone:
case <-ctx.Done():
    select {
    case <-pusherDone:
    case <-time.After(8 * time.Second):
        ...
    }
}
```

If `ctx.Done()` fires but the pusher goroutine hangs (e.g., `pushDraft` blocks on an HTTP call that never returns), the fallback timeout of 8 seconds prevents deadlock. However, `pushDraft` acquires `draftMu` mutex — if the pusher goroutine is blocked on `draftMu` while the main goroutine is waiting for `pusherDone`, and the main goroutine holds `draftMu` (indirectly via `endStream` -> `draftMu.Lock()`), this would deadlock. Let me check:

The main goroutine (stream.go:392, the SSE consumer goroutine) calls `signalFlush()` in its defer (line 396), which acquires `draftMu.Lock()` → `a.dismissSessionDraft(chatID)` → `draftMu.Unlock()`. The SSE consumer goroutine calls `signalFlush()` → acquires and releases `draftMu`, then closes `finished`. The pusher then acquires `draftMu`. This is a correct producer-consumer handoff. **No deadlock risk here.** ✅

### P3 — serve_backend.go:404 (else branch): tmpf.Close() called twice in some paths

Looking more carefully: if WriteString succeeds, `tmpf.Close()` at line 398 closes the file. Then Chmod is called on `tmpf.Name()`. If Chmod fails, we fall through to line 404 (`else` block of `if err == nil` on line 396). The `else` branch at line 403-404 closes `tmpf` again (double close) — this is benign in Go (returns an error which is ignored), but it means the temp file is never cleaned up because `keyFile` is empty.

### 4.3 Memory — media.go LRU cache

### P1 — media.go:122-183: cleanupCacheDir race condition

```go

    dir := a.cacheDir(category, chatID)
    entries, err := os.ReadDir(dir)

    // Global LRU check
```

- **Race**: `cleanupCacheDir` is called from `downloadTelegramFile` (media.go:115) **without holding any lock**. Two concurrent downloads for different chats/categories could both call `walkCacheFiles` simultaneously, collect file lists, and then both delete files. This could lead to:

### P1 — extractVideoFrames and convertPDFToImages: no memory limit

### 4.4 Subprocess management

### P1 — serve_backend.go:431-441: cmd.Wait goroutine leak risk

```go
go func() {
    err := cmd.Wait()
    s.mu.Lock()
    if s.serveCmd == cmd {
        s.serveCmd = nil
    }
    s.mu.Unlock()
    ...

```

If `startServe` returns an error from `waitServeReady` (line 443), it calls `a.stopServe(chatID)` which SIGTERM's the process. ## 5. GRACEFUL SHUTDOWN

### 5.1 SIGTERM/SIGINT handling (main.go:482-547)

The shutdown sequence:
1. `ctx.Done()` fires via signal.NotifyContext
2. `close(app.healthCheckStop)` → stops health check goroutine
3. `app.msgWg.Wait()` → waits for active message/callback handlers
4. `app.cancelAllTasks()` → cancels all running turns via context cancellation
5. `app.waitTasksDone(5 * time.Second)` → waits up to 5s for tasks to finish
6. `app.stopAllServes()` → SIGTERM all serve processes, wait 5s, SIGKILL
7. Stop sentTextCache debounce timer, flush cache to disk
8. Stop cron manager cron + watcher

### P1 — main.go:514: msgWg.Wait() called before cancelAllTasks

`msgWg.Wait()` waits for `handleMessage` and `handleCallbackQuery` goroutines. But these handlers call `runTask` internally, which spawns more goroutines. The `msgWg.Wait()` on line 514 only waits for the handler to finish dispatching, not for the `runTask` goroutines themselves (those are tracked by `s.task`). This is **correct** — the handler goroutine returns quickly after calling `runTask` (the actual work is in the goroutines spawned by `runTask`). The tasks are waited for in step 4+5.

### P1 — main.go:516-517: waitTasksDone(5s) may not be enough

```go
app.waitTasksDone(5 * time.Second)
```

If a task is in the middle of a long SSE stream or a slow HTTP request, 5 seconds may not be enough. The context cancellation in step 4 will propagate to `consumeServeEvents` (which checks `ctx.Done()`), but if the SSE read is blocking on `sc.Scan()` (bufio.Scanner), it won't unblock until the server sends more data or the TCP connection times out (default TCP timeout is minutes). The SSE idle watchdog (serve_backend.go:796-812) fires after 300 seconds of inactivity, which is too long.

However, `cancelAllTasks()` sets the context cancellation, which triggers `closeBody()` via the goroutine at serve_backend.go:773-779, which closes the SSE connection, which causes `sc.Scan()` to return false, which unblocks the consumer. So the chain is:

1. `cancelAllTasks()` → context cancel
2. ctx.Done() fires in connectAndConsumeSSE (goroutine at line 773)
3. `closeBody()` closes resp.Body
4. `sc.Scan()` returns false → loop exits → eventsDone receives result
5. runServeTurn returns → runTask defer clears s.task

This should work within seconds. **But the goroutine at line 773 runs `select { case <-ctx.Done(): closeBody() ... }`. The defer `close(ctxDoneChan)` happens when `connectAndConsumeSSE` returns. If the scanner loop is still running, `ctxDoneChan` close races with `ctx.Done()` — but both paths call `closeBody()` via `sync.Once`, so double-close is safe.**

### P2 — main.go:526-531: cronManager Stop may block

```go
if app.cronManager != nil {
    app.cronManager.cron.Stop()
    if app.cronManager.watcher != nil {
        app.cronManager.watcher.Close()
    }
}
```

`cron.Stop()` returns a `context.Context` that completes when all running jobs finish. The return value is **discarded** — we don't wait for it. This means a cron job that's currently running (e.g., `triggerCronTask`) will continue to run during shutdown. If it's in the middle of `cmd.CombinedOutput()` (cron.go:290), it could block indefinitely. The cron job has a 10-minute timeout context (cron.go:269), so it will eventually complete, but not before the bridge process exits (which happens on `return` from main).

### P2 — cron.go:82: watchLoop goroutine not stopped on shutdown

The `CronManager.watchLoop()` goroutine (cron.go:152) runs until `cm.watcher.Events` or `cm.watcher.Errors` is closed. In shutdown (main.go:528-531), `cm.watcher.Close()` is called, which closes the Events and Errors channels, causing `!ok` to be true and the watchLoop to return. This is correct. ✅

### 5.2 Subprocess cleanup

### ✅ serve_backend.go:262-279: stopSessionServe with SIGTERM → 5s timeout → SIGKILL → 2s timeout

The termination pattern:
1. `syscall.Kill(-cmd.Process.Pid, syscall.SIGTERM)` (negative PID = process group)
2. Wait up to 5s in goroutine + select
3. If timeout, `SIGKILL`
4. Wait additional 2s for port release

This is a correct, battle-tested pattern. ✅

### ✅ stale_serve.go: terminateProcessGroup has PID reuse protection

Uses `readProcStartTime` to detect PID recycling, preventing killing a new unrelated process. ✅

### P2 — serve_backend.go:110-118: Debounce timer in pusher goroutine

```go
debounce := time.NewTimer(time.Hour)
debounce.Stop()
```

The pusher goroutine creates a timer with a 1-hour initial duration, immediately stops it, and then uses `debounce.Reset(streamDebounce)` throughout its lifetime. When the goroutine exits (via `return` at stream.go:812), the timer is abandoned — it's never explicitly stopped. However, `Reset` on an already-fired timer returns false (draining the channel), and a stopped timer can be GC'd. The `close(pusherDone)` happens before the goroutine exits, but the timer is not explicitly stopped. In practice, Go's GC will collect the timer, and since `debounce.C` is unbuffered (standard time.Timer), no one will ever read from it after the goroutine ends. **Low risk** — the timer can fire once harmlessly (no one reads the channel). The ticker inside `debounce` itself is not a ticker, it's a `time.Timer`, so it's a one-shot event.

Actually, `time.NewTimer` creates a timer with a channel of capacity 1. Even after the goroutine exits, if the timer fires and no one reads it, it's just a goroutine in the Go runtime waiting to send on a channel. The runtime eventually cleans up. **Not a leak in practice** — Go 1.14+ timers are efficient.

## SUMMARY BY RISK LEVEL

### 🔴 P0 — Critical (fix immediately)

1. **restart.go:254 — Restart watchdog goroutine never exits, ticker never stopped**  
   The `startRestartWatchdog` goroutine runs forever with no shutdown mechanism. The `time.NewTicker(15s)` is never stopped. During SIGTERM shutdown, this goroutine continues running, potentially racing with state changes.  
   **Fix**: Add a shutdown channel/context so the goroutine returns on shutdown. Defer `ticker.Stop()`.

2. **restart.go:238 — Graceful restart goroutine races with SIGTERM shutdown**  
   The `/restart` command spawns a goroutine that calls `cancelAllTasks()`, `waitTasksDone(30s)`, `stopAllServes()` concurrently with the same calls in the SIGTERM handler.  
   **Fix**: Integrate the restart flow with the shutdown context, or set a flag that prevents concurrent shutdown.

### P1 — High

1. **main.go:539-544 — Unbounded goroutine creation on updates**  
   Every Telegram update spawns a new goroutine with no limit. Under flood, this can create thousands of goroutines.  
   **Fix**: Add a semaphore to cap concurrent handler goroutines.

2. **media.go:122-183 — Cache cleanup race**  
   `cleanupCacheDir` runs without any lock, allowing concurrent cleanup operations to interfere.  
   **Fix**: Add a global `cacheMu sync.Mutex` for cleanup operations.

3. **main.go:514 — msgWg.Wait() before cancelAllTasks**  
   While functionally correct, the order means handler goroutines that just spawned tasks complete before those tasks are cancelled. A handler that starts a new task after cancellation could leave it running.  
   **Fix**: Cancel tasks first (`cancelAllTasks`), then wait for handles (`msgWg.Wait()`).

### P2 — Medium

1. **handler.go:134-212 / callback.go:552-599 — pendingClarify concurrent mutation**  
   Multiple goroutines can mutate `pendingClarify` state across lock/unlock boundaries.  
   **Fix**: Serialize clarify state access with a dedicated mutex or hold the session lock across the entire operation.

2. **serve_backend.go:507 — TOCTOU race on serveCmd.ProcessState**  
   ProcessState read outside the lock after cmd pointer was read under lock.  
   **Fix**: Read all necessary fields (including ProcessState) under the lock, or use `Process.Signal(0)` to check liveness.

3. **cron.go:289 — triggerCronTask long-running blocking call during shutdown**  
   `exec.CommandContext(ctx, ...).CombinedOutput()` can block for up to 10 minutes, and `cron.Stop()` return value is discarded.  
   **Fix**: Track running cron jobs and wait for them during shutdown.

4. **stream.go:839-856 — Error reply sent after shutdown**  
   In `runTask`, if the context is cancelled during shutdown, it replies with an error message to the user. This is a minor UX issue but also wastes shutdown time.  
   **Fix**: Check if we're shutting down before sending error replies.
